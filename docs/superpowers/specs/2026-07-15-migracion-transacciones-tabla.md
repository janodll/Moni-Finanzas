# Plan: Migrar transacciones del blob JSON a una tabla relacional en Supabase (Fase 1)

**Objetivo:** que las **transacciones** vivan en una tabla real de Supabase (una fila por transacción, `id` autogenerado, `nro_operacion` único). Esto elimina de raíz la pérdida de datos por escrituras que se pisan y habilita deduplicación a nivel de base de datos.

**Alcance (Fase 1):** SOLO `transacciones`. Todo lo demás (cuentas, tarjetas, categorías, metas, recordatorios, presupuestos, configuración, `transacciones_pendientes`, `trabajos_pendientes`) **se queda en el blob `moni_state`** tal como está hoy.

**Modo de trabajo:** Jano ejecuta paso a paso. Los pasos marcados con ⚠️ tocan datos reales o el despliegue — hazlos con cuidado y verifica antes de seguir. Si algo no calza, para y consulta antes de continuar.

**Regla de oro de la arquitectura nueva:**
- El **blob `moni_state`** guarda TODO **menos** `transacciones`.
- La **tabla `transacciones`** guarda las transacciones.
- Al leer, el servidor une ambos y entrega el mismo formato de siempre (el dashboard no nota el cambio al leer).
- Al escribir transacciones, se usa la tabla (INSERT/UPDATE/DELETE), nunca el blob.

## Parte 2 — Copiar tus transacciones actuales del blob a la tabla ⚠️ (backfill, una sola vez)

Este script lee el blob actual de Supabase, inserta cada transacción en la tabla y ajusta la secuencia de `id`. **Es idempotente por `nro_operacion`, pero NO lo corras dos veces** sin revisar (las transacciones sin `nro_operacion` se duplicarían).

1. Crea `scratch/backfill_transacciones.mjs` con este contenido:

```javascript
// Uso: SUPABASE_URL=... SUPABASE_KEY=... node scratch/backfill_transacciones.mjs
const SUPABASE_URL = process.env.SUPABASE_URL;
const SUPABASE_KEY = process.env.SUPABASE_KEY;
if (!SUPABASE_URL || !SUPABASE_KEY) { console.error("Falta SUPABASE_URL/SUPABASE_KEY"); process.exit(1); }

const H = { apikey: SUPABASE_KEY, Authorization: `Bearer ${SUPABASE_KEY}`, 'Content-Type': 'application/json' };

// Columnas válidas de la tabla (cualquier otro campo del blob se ignora)
const COLS = ['id','fecha','tipo','categoria','descripcion','monto','moneda','cuenta_id',
  'tarjeta_id','fijo','nro_operacion','transfer_id','banco_o_metodo','descripcion_original',
  'telegram_message_id','created_at'];

const pick = (t) => {
  const o = {};
  for (const c of COLS) if (t[c] !== undefined && t[c] !== null) o[c] = t[c];
  if (o.fecha === undefined) o.fecha = new Date().toISOString().slice(0,10);
  if (o.tipo === undefined) o.tipo = 'GASTO';
  return o;
};

async function main() {
  // 1. Leer el blob
  const res = await fetch(`${SUPABASE_URL}/rest/v1/moni_state?select=data&id=eq.1`, { headers: H });
  const rows = await res.json();
  const state = rows?.[0]?.data;
  const txs = state?.transacciones || [];
  console.log(`Transacciones en el blob: ${txs.length}`);
  if (txs.length === 0) { console.log("Nada que migrar."); return; }

  // 2. Confirmar que la tabla está vacía (seguridad)
  const cnt = await fetch(`${SUPABASE_URL}/rest/v1/transacciones?select=id`, { headers: { ...H, Prefer: 'count=exact', Range: '0-0' } });
  const total = cnt.headers.get('content-range')?.split('/')?.[1];
  if (total && total !== '0') { console.error(`La tabla YA tiene ${total} filas. Aborta para no duplicar.`); process.exit(1); }

  // 3. Insertar en lotes de 100
  const payload = txs.map(pick);
  for (let i = 0; i < payload.length; i += 100) {
    const chunk = payload.slice(i, i + 100);
    const r = await fetch(`${SUPABASE_URL}/rest/v1/transacciones`, {
      method: 'POST', headers: { ...H, Prefer: 'return=minimal' }, body: JSON.stringify(chunk)
    });
    if (!r.ok) { console.error(`Fallo lote ${i}:`, r.status, await r.text()); process.exit(1); }
    console.log(`Insertadas ${Math.min(i+100, payload.length)}/${payload.length}`);
  }

  // 4. Verificación de conteo
  const cnt2 = await fetch(`${SUPABASE_URL}/rest/v1/transacciones?select=id`, { headers: { ...H, Prefer: 'count=exact', Range: '0-0' } });
  const total2 = cnt2.headers.get('content-range')?.split('/')?.[1];
  console.log(`Filas en la tabla ahora: ${total2} (esperado ${txs.length})`);
  console.log("Listo. AHORA corre en el SQL Editor de Supabase el ajuste de secuencia (ver Parte 2, paso 3).");
}
main().catch(e => { console.error(e); process.exit(1); });
```

## Parte 3 — Cambios en el servidor (`server.js`)

### 3.1 Nuevo módulo de acceso a la tabla

Crea `db.js` en la raíz del proyecto:

```javascript
// db.js — acceso a la tabla de transacciones en Supabase (PostgREST, sin librerías nuevas)
const SUPABASE_URL = process.env.SUPABASE_URL;
const SUPABASE_KEY = process.env.SUPABASE_KEY;

function headers(extra = {}) {
  return { apikey: SUPABASE_KEY, Authorization: `Bearer ${SUPABASE_KEY}`, 'Content-Type': 'application/json', ...extra };
}
const BASE = () => `${SUPABASE_URL}/rest/v1/transacciones`;

// Trae todas las transacciones (más nuevas primero, como el unshift de antes)
export async function dbGetTransacciones() {
  const r = await fetch(`${BASE()}?select=*&order=id.desc`, { headers: headers() });
  if (!r.ok) throw new Error(`dbGetTransacciones ${r.status}: ${await r.text()}`);
  return await r.json();
}

// Inserta una transacción. Devuelve la fila creada (con id). Si choca con el
// índice único de nro_operacion (409), devuelve null (duplicado ignorado).
export async function dbInsert(tx) {
  const r = await fetch(BASE(), {
    method: 'POST', headers: headers({ Prefer: 'return=representation' }), body: JSON.stringify(tx)
  });
  if (r.status === 409) { console.log('[db] Duplicado por nro_operacion, ignorado:', tx.nro_operacion); return null; }
  if (!r.ok) throw new Error(`dbInsert ${r.status}: ${await r.text()}`);
  const rows = await r.json();
  return rows[0];
}

// Inserta un par de transferencia/pago (gasto + ingreso) enlazadas por transfer_id.
export async function dbInsertPair(gasto, ingreso) {
  const g = await dbInsert(gasto);
  if (!g) return { gasto: null, ingreso: null };
  await dbUpdate(g.id, { transfer_id: g.id });
  const i = await dbInsert({ ...ingreso, transfer_id: g.id });
  return { gasto: { ...g, transfer_id: g.id }, ingreso: i };
}

export async function dbUpdate(id, fields) {
  const r = await fetch(`${BASE()}?id=eq.${id}`, {
    method: 'PATCH', headers: headers({ Prefer: 'return=representation' }), body: JSON.stringify(fields)
  });
  if (!r.ok) throw new Error(`dbUpdate ${r.status}: ${await r.text()}`);
  const rows = await r.json();
  return rows[0] || null;
}

export async function dbDelete(id) {
  const r = await fetch(`${BASE()}?id=eq.${id}`, { method: 'DELETE', headers: headers() });
  if (!r.ok) throw new Error(`dbDelete ${r.status}: ${await r.text()}`);
}

export async function dbDeleteByTransfer(transferId) {
  const r = await fetch(`${BASE()}?transfer_id=eq.${transferId}`, { method: 'DELETE', headers: headers() });
  if (!r.ok) throw new Error(`dbDeleteByTransfer ${r.status}: ${await r.text()}`);
}
```

Al inicio de `server.js`, importa:
```javascript

```

app.delete('/api/transaccion/:id', requireLocalAuth, async (req, res) => {

### 4.1 `state.js` — dejar de mandar transacciones en el guardado global

En `saveState()` (~257), el POST a `/api/data` debe mandar el estado **sin** `transacciones`:

// ...body: JSON.stringify(payload)

Agrega helpers de API en `state.js` (para reusar en los handlers):
```javascript
async function txApi(url, method, body) {
  const r = await fetch(`${API_BASE}${url}`, {
    method, headers: { 'Content-Type': 'application/json', 'X-Local-Token': LOCAL_TOKEN },
    body: body ? JSON.stringify(body) : undefined
  });
  if (!r.ok) throw new Error('Error de red al guardar la transacción');
  return await r.json();
}
export async function apiAddTransaccion(tx)      { return (await txApi('/api/transaccion', 'POST', tx)).data; }
export async function apiAddPar(gasto, ingreso)  { return (await txApi('/api/transaccion/par', 'POST', { gasto, ingreso })).data; }
export async function apiUpdateTransaccion(id, f){ return (await txApi(`/api/transaccion/${id}`, 'PUT', f)).data; }
export async function apiDeleteTransaccion(id, transferId) { const q = transferId ? `?transfer_id=${transferId}` : ''; return txApi(`/api/transaccion/${id}${q}`, 'DELETE'); }
```

### 4.2 Convertir cada handler de mutación

Patrón general: donde antes hacías `state.transacciones.unshift(tx); saveState();`, ahora:
```javascript

```
- En modo `isLocalStorageMode`, conserva el comportamiento viejo (unshift local + saveState) para que el modo offline siga andando. Ej: `if (isLocalStorageMode) { state.transacciones.unshift(tx); saveState(); } else { const saved = await apiAddTransaccion(tx); if (saved) state.transacciones.unshift(saved); updateUI(); }`.

Sitios (por tipo):

- **Alta de transacción simple** — `main.js:651` (form transacción), `main.js:974` (otro ingreso): usar `apiAddTransaccion`.
- **Lote** — `main.js:417` (varias `nuevas`): recórrelas con `apiAddTransaccion` (o crea un endpoint batch si prefieres; opcional).
- **Transferencia** — `main.js:696/710` (txGasto + txIngreso): usar `apiAddPar(txGasto, txIngreso)` en vez de dos unshift. Con la respuesta, `unshift` ambas piernas devueltas.
- **Pago de recordatorio** — `main.js:874/887` (tarjeta: gasto+ingreso) y `main.js:900` (servicio): las transacciones por `apiAddPar`/`apiAddTransaccion`; el cambio del **recordatorio** (estado/fecha) sigue por `saveState()` (es config/blob).
- **Aporte/retiro de meta** — `main.js:814/833`: transacción por `apiAddTransaccion`; el cambio de la **meta** por `saveState()`.
- **Editar transacción** — `history.js` (edición): `await apiUpdateTransaccion(id, campos)` y refresca la fila local con la respuesta, en vez de mutar el array + `saveState()`.
- **Borrar transacción** — `history.js:353` (simple): `await apiDeleteTransaccion(id)`; el "deshacer" re-crea con `apiAddTransaccion`.
- **Borrar transferencia** — `history.js:339-347` (borra ambas piernas por `transfer_id`): `await apiDeleteTransaccion(tx.id, tx.transfer_id)`; el "deshacer" re-crea el par.

## Parte 7 — Rollback (si algo sale mal)

1. **Revertir el servidor** al deploy anterior: en git, vuelve a `main` previo (o `git revert` del merge) y push → Render redespliega la versión blob.
2. La versión vieja lee las transacciones del **blob**, que sigue **intacto** (nunca lo borramos; solo dejamos de escribirle transacciones). No se pierde nada de lo previo al corte.
3. **Única salvedad:** las transacciones que hayan entrado a la **tabla** después del corte no están en el blob. Por eso el corte se verifica rápido. Si hubo pocas, se re-registran a mano; o antes de revertir, expórtalas de la tabla y agrégalas al backup.
4. Si prefieres, guarda el `Moni_Backup` del Pre-flight y reimpórtalo tras revertir para volver al estado exacto de antes.

## Notas y decisiones

- **`transacciones_pendientes` se queda en el blob** en esta fase (es chica y transitoria; el mutex la cubre). Migrarla es opcional en una fase futura.
- **`id`**: lo asigna la base (identity). Se acabó el `Math.max(...)` (que además podía reventar el stack) y las colisiones de id.
- **Dedup real**: el índice UNIQUE de `nro_operacion` es la garantía; el código deja de ser el responsable.
- **Sin librerías nuevas**: todo por PostgREST vía `fetch`, igual que ya usas.
- **Modo offline (localStorage)**: se conserva el comportamiento viejo (transacciones en localStorage) para no romper el fallback sin conexión.
- **Fase 2 (futuro, opcional)**: migrar cuentas/tarjetas/etc. a tablas y mover el cálculo de saldos a SQL/servidor. No es necesario para resolver la pérdida de datos.
